# AIS-0000 — AI-OPS Standard Introduction

## 1. Purpose

AI-OPS is an open, vendor-neutral standard for safe, repeatable and auditable AI operations on infrastructure and software systems.

AI-OPS defines the required behavior of AI agents before, during and after an operation that can inspect, modify or control a target system. It establishes a common lifecycle, reporting model and compliance baseline that can be implemented by different agents, runtimes and automation platforms.

## 2. Scope

AI-OPS applies to AI-assisted operations involving:

- Linux hosts and virtual machines;
- containers and orchestration platforms;
- systemd services and scheduled jobs;
- network services, ports, firewalls and reverse proxies;
- storage, filesystems and backup systems;
- source repositories and CI/CD systems;
- databases and API services;
- OpenAPI, TypeSpec and generated SDK workflows;
- MCP servers and MCP clients;
- AI coding agents and local AI gateways;
- monitoring, logging and observability systems;
- local networks, VPNs and Tailscale environments;
- industrial and edge systems where deterministic boundaries must be preserved.

## 3. Problem statement

AI agents frequently operate with incomplete, stale or assumed context. Common failure modes include:

- assuming that a runtime, package or service is not installed;
- selecting a port that is already in use;
- creating a duplicate service instead of discovering an existing one;
- overwriting configuration without a backup;
- changing a large part of the environment to solve a narrow problem;
- modifying the wrong host, repository, branch or environment;
- treating successful command execution as proof of a correct result;
- failing to record changed files, commands and decisions;
- providing no tested rollback path;
- exposing secrets in logs, reports or generated files;
- giving probabilistic AI components direct control over safety-critical logic.

AI-OPS addresses these failures by defining mandatory operational behavior rather than prescribing a particular product.

## 4. Goals

AI-OPS is designed to provide:

1. **Safety** — protect the existing system and prevent avoidable damage.
2. **Evidence-based operation** — use discovered state instead of assumptions.
3. **Minimal change** — satisfy the approved objective with the smallest practical modification.
4. **Repeatability** — make operations reproducible across agents and environments.
5. **Auditability** — preserve machine-readable and human-readable evidence.
6. **Backup and rollback** — make state-changing work recoverable.
7. **Verification** — test the resulting state, not merely command completion.
8. **Human control** — require approval for destructive, high-risk and safety-critical actions.
9. **Vendor neutrality** — support different models, agents, MCP implementations and infrastructure tools.
10. **Local-first operation** — support systems that must function without public cloud dependency.
11. **Multi-agent interoperability** — provide a common operational contract for Kimi Code, Claude Code, Codex, OpenCode and other agents.
12. **Compliance testing** — make requirements independently testable.

## 5. Non-goals

AI-OPS is not:

- a configuration management system such as Ansible;
- an infrastructure provisioning system such as Terraform;
- a container runtime or orchestrator;
- a monitoring, backup or security product;
- an MCP server or protocol replacement;
- an AI model or coding agent;
- a universal command library;
- a replacement for authorization, RBAC or secret management;
- a replacement for deterministic control or safety logic.

AI-OPS MAY be implemented through existing tools, but it does not require a particular implementation.

## 6. Target audience

The intended audience includes:

- Linux administrators;
- DevOps and platform engineers;
- site reliability engineers;
- AI and automation engineers;
- MCP server and client developers;
- security engineers;
- industrial automation and edge-computing teams;
- maintainers of AI coding agents;
- open-source project maintainers.

## 7. Standard architecture

AI-OPS consists of four layers:

```text
AI-OPS Standard
├── Specification
├── Reference Implementation
├── Compliance Suite
└── Dashboard
```

### 7.1 Specification

The specification defines normative behavior, data models, lifecycle requirements and compliance criteria.

### 7.2 Reference Implementation

The reference implementation provides scripts, libraries and examples that demonstrate one valid implementation of the standard.

### 7.3 Compliance Suite

The compliance suite verifies whether an agent or implementation satisfies specific AI-OPS requirements.

### 7.4 Dashboard

The dashboard presents discovered system state, operation history, agent capabilities, compliance status and verification results.

## 8. Operational lifecycle

A conforming state-changing operation follows this lifecycle:

```text
Human request
    ↓
Target identification
    ↓
Discovery
    ↓
Analysis
    ↓
Plan
    ↓
Risk classification
    ↓
Approval when required
    ↓
Backup
    ↓
Execution
    ↓
Verification
    ↓
Rollback decision
    ↓
Audit report
```

Skipping a mandatory phase is permitted only when the relevant specification explicitly defines an exception.

## 9. Core actors

### 9.1 Human operator

The human operator defines the objective and remains the final authority for destructive, high-risk, security-sensitive and safety-critical operations.

### 9.2 AI agent

The AI agent interprets the objective, discovers the environment, prepares a plan, executes approved work and produces evidence.

### 9.3 Target system

The target system is the host, repository, service, cluster, API, device or environment being inspected or changed.

### 9.4 Policy and approval layer

The policy layer classifies risk, constrains permissions and determines whether human approval is required.

### 9.5 Compliance evaluator

The compliance evaluator verifies operation reports and implementation behavior against normative requirements.

- `specifications/` contains normative AIS documents.
- `implementations/` contains non-normative reference code.
- `compliance/` contains test cases, fixtures and conformance profiles.
- `schemas/` contains machine-readable schemas.
- `templates/` contains reusable reports and plans.
- `scripts/` contains project maintenance utilities.
- `dashboard/` contains the dashboard implementation.
- `skills/` contains agent-readable operational instructions.
- `examples/` contains non-normative examples.
- `rfcs/` contains proposals that are not yet part of the accepted standard.
- `docs/` contains explanatory documentation.

## 11. Document families

The project defines:

- `AIS-STYLE-*` — editorial and document governance rules;
- `AIS-0000` and later — normative standard documents;
- `RFC-*` — proposed features or changes before standardization;
- schemas — machine-readable normative or supporting models;
- implementation guides — non-normative operational guidance.

## 12. Normative language

The terms **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY** and **OPTIONAL** are interpreted according to `AIS-STYLE-0001`.

Normative requirements MUST have stable identifiers and MUST be independently testable.

## 13. Language policy

English is the canonical language of the standard.

Serbian translations are official project translations but are non-canonical. A translation MUST NOT introduce, remove or alter normative requirements. When language versions conflict, the canonical English document takes precedence.

## 14. Versioning

The project and individual specifications use semantic versioning:

- MAJOR — incompatible normative change;
- MINOR — backward-compatible normative addition;
- PATCH — clarification without behavioral change.

Document status and document version are independent. A Draft document may have multiple versions before acceptance.

## 15. Security and safety model

AI-OPS assumes that AI output can be incomplete or incorrect.

Therefore:

- authorization MUST be enforced outside the model;
- secrets MUST be redacted from reports and logs;
- destructive operations MUST be explicitly classified;
- safety-critical control MUST remain deterministic;
- approvals MUST be bound to a specific plan, target and operation;
- verification MUST use observed system state;
- rollback MUST be prepared before execution when state changes are involved.

## 16. Compliance model

Compliance is requirement-based, not document-based.

An implementation MAY support only a defined profile, but it MUST clearly declare:

- implemented AIS documents;
- implemented requirement identifiers;
- unsupported requirements;
- deviations and extensions;
- compliance test results;
- implementation and agent versions.

Self-declared compliance without evidence is non-conforming.

## 18. Conformance requirements

> **AIS-0000-REQ-001:** A conforming implementation MUST identify the target system before performing discovery or execution.

> **AIS-0000-REQ-002:** A conforming state-changing operation MUST include discovery, planning, backup, execution, verification and audit phases unless an applicable AIS document defines a documented exception.

> **AIS-0000-REQ-003:** A conforming implementation MUST produce a machine-readable operation report.

> **AIS-0000-REQ-004:** A conforming implementation MUST preserve human authority over high-risk, destructive and safety-critical operations.

> **AIS-0000-REQ-005:** A conforming implementation MUST NOT treat AI-generated output as a replacement for deterministic authorization or safety controls.
